# PROTOCOL APPLICATION MATRIX — direct-supply-meal

**Phase:** 2 of 4 (Workflow Design)
**Status:** Draft, reviewable artifact

## Purpose

Phase 1 catalogued 18 DuloCore protocols worth porting to ds-meal and named a graduation seam for each (see `/root/.claude/plans/replicated-plotting-rivest.md`, "PROTOCOL CATALOG"). This document binds each abstract protocol to a concrete home in ds-meal: the specific file where it lives, the specific UI moment where a reviewer sees it running, and the exact function or import line where App Phase 2 extends it without rewrite.

A reviewer should be able to read this doc top-to-bottom and, for every protocol, answer three questions in under a minute:

1. **Where does this protocol physically live in the repo?**
2. **At what moment during the demo is it visibly active?**
3. **If we graduated this to production tomorrow, which line of code would we change?**

The Two-Horizon Rule applies everywhere: every section ends with the named Phase 2 seam.

### P1 — Agent Hierarchy (ATOMIC-S)
**Phase 1 application in ds-meal:** Collapsed 4-tier ATOMIC-S to 2 tiers. L1 MEAL Director lives in `agents/drivers/menu_planner.py` and `agents/drivers/nl_ordering.py`, thin `ClaudeSDKClient` wrappers on Sonnet. L3 Workers are `AgentDefinition` subagents on Haiku inside those same drivers. Concurrency cap of ≤4 workers enforced by the driver. Mandatory-delegation rule ("L1 never touches a DB") honored because drivers only call `invoke_tool()`.
**When it runs:** The moment a signed-in admin at `/agents/menu-plan` hits "Plan week." The Director decomposes "plan Riverside breakfasts for week-of-2026-04-27" and fans out to three Worker subagents.
**Phase 2 seam:** `agents/drivers/dispatch()` body swaps from in-process `asyncio.gather` to Inngest event emit; driver files stay untouched.

### P2 — Task/Exploration Depth Evaluation
**Phase 1 application in ds-meal:** Pure function `score_query(text) -> (level, n_agents, shape)` in `agents/depth_scorer.py`. Implements the 6-dimension rubric. Called at the top of `nl_ordering.run()` and `menu_planner.run()`. No hook, no enforcement — just a function that sets `n_agents` for fan-out.
**When it runs:** When a dietitian types "plan Riverside diabetic-friendly breakfasts for next week" into `/agents/menu-plan`. The scorer returns `(deep, 4, parallel)` and the Director spawns four Workers.
**Phase 2 seam:** `agents/depth_scorer.py::score_query()` gains a `PreToolUse` hook wrapper in `.claude/hooks/depth_enforcer.py` that blocks under-scoped dispatches — function signature unchanged.

### P3 — Progressive Disclosure (3-file doc spine)
**Phase 1 application in ds-meal:** `CLAUDE.md` ≤120 lines, three items: project index, progressive-disclosure pattern, task-depth pointer. `docs/INDEX.md` topic router with 3 buckets. `docs/CATALOG.md` feature-status table, one row per protocol. CATALOG and INDEX never duplicate detail — link only.
**When it runs:** Every time a reviewer opens the repo. First click → `CLAUDE.md`, one hop → `INDEX.md`, second hop → system doc. No file over 120 lines for the entry-point trio.
**Phase 2 seam:** `docs/CATALOG.md` gains a "Status" column auto-generated by `scripts/catalog_status.py`; 3-file spine unchanged.

### P4 — Claude Agent SDK Integration (Route Rule)
**Phase 1 application in ds-meal:** All 9 `@tool` functions in `agents/tools_sdk.py`. FastAPI routes in `app/routes/*.py` import `invoke_tool` and never import SQLModel sessions. Sibling `agents/tools.py` holds pure async DB helpers (no decorator).
**When it runs:** Every time the user submits an NL order at `/orders/new`. The route calls `invoke_tool(resolve_recipe, ...)` → `invoke_tool(scale_recipe, ...)` → `invoke_tool(schedule_order, ...)`. Routes never see a DB session.
**Phase 2 seam:** `agents/tools_sdk.py` stays identical. Routes migrate from in-process `invoke_tool()` to MCP-transport `invoke_tool()`. One import line per route.

### P5 — Director System (single L1 MEAL Director)
**Phase 1 application in ds-meal:** One L1 Director named `MEAL`, instantiated per-request inside the two drivers. No L0, no L2, no spawn script, no message bus. Session banner format documented in `docs/systems/director-system/DIRECTOR-ARCHITECTURE.md` but only emitted to `logs/agent_trace.jsonl`.
**When it runs:** On any POST to `/agents/menu-plan` or `/agents/nl-order`. Driver logs a one-line banner (`[MEAL/L1] start trace=... user=admin@dulocore.com`) before fanning out Workers.
**Phase 2 seam:** `agents/drivers/__init__.py::get_director()` becomes a factory resolving directors from a registry table; drivers unchanged.

### P6 — Karpathy Wiki Knowledge Base (3-layer memory)
**Phase 1 application in ds-meal:** Three-layer shape. Layer 1 = SQLite `agent_trace` table (immutable), written by `agents/observability.py`. Layer 2 = `wiki/topics/{agent_name}/*.md` (Markdown + YAML frontmatter), written by `wiki/compiler.py` on `make compile-wiki`. Layer 3 = `wiki/schema.yaml`. `wiki/TOPICS-INDEX.md` is the sole retrieval surface. Seeded with 20–30 synthetic traces.
**When it runs:** Right before any agent call — `agents/drivers/*.py` reads `wiki/TOPICS-INDEX.md` and injects relevant topic pages into the Director's system prompt.
**Phase 2 seam:** `wiki/compiler.py::cluster_traces()` replaces hand-grouping with MiniLM + vector clustering; topic-page format unchanged.

### P7 — Karpathy Auto-Research (cross-task learning loop)
**Phase 1 application in ds-meal:** Two altitudes. (a) Intra-task self-healing = SDK-native retry + Haiku→Sonnet escalation, configured in each driver. (b) Cross-task learning = `agents/observability.py::record_outcome()` appends to `agent_trace` + JSONL → `wiki/compiler.py` synthesizes patterns into topic pages → next session's drivers read them via P6's injection hook. Demo showcase: `wiki/topics/nl_ordering/recipe-aliases.md` learns "oats" means "Overnight Oats" for Riverside.
**When it runs:** After each NL-order submission a new trace row writes. On `make compile-wiki` the compiler regenerates the aliases page; next order that says "oats" resolves faster.
**Phase 2 seam:** `make compile-wiki` becomes a systemd 24-hour timer; `wiki/compiler.py` entry point unchanged. **Additional Phase 2 seam:** `wiki/graph.py` introduces a graph KB (Kuzu / Neo4j / NetworkX) for gap detection — see `KARPATHY-AUTO-RESEARCH-WORKFLOW.md` §12.

### P8 — Progressive Memory (SDK memory layer, 4 memory types)
**Phase 1 application in ds-meal:** No separate PMO tier. ds-meal's "memory" IS the wiki, with 4 memory types: `feedback`, `project`, `reference`, `user`. Hand-written seed memories in `memory/` — five files with YAML frontmatter. `memory/MEMORY.md` is the ≤50-line index.
**When it runs:** At session-start, Director prompt includes the 5 seed memory files verbatim. During the demo, the dietary-compliance tool cites `feedback_dietary_rules.md` in its reasoning.
**Phase 2 seam:** `wiki/schema.yaml` grows from 4 memory types to 6 (adds `decision`, `observation`) — no code change, config only.

### P9 — 5-Stage Dev Workflow + 5-Stage DoD (collapsed)
**Phase 1 application in ds-meal:** Dev workflow collapsed to 3 stages (Plan → Code → Verify) in `docs/systems/verification-stages/VERIFICATION-STAGES.md`. DoD is 5-stage: Unit → Integration → E2E → Deployed → RealAuth. UserAccessible skipped. Magnitude table in `CLAUDE.md` as a reference link.
**When it runs:** Every PR/commit. A "medium" change requires Unit + Integration + E2E green before merge; RealAuth tests run on `main` against staging.
**Phase 2 seam:** `docs/systems/verification-stages/` expands from 3 stages back to 5; CI adds a `UserAccessible` Playwright suite.

### P10 — Hook System (one reference hook)
**Phase 1 application in ds-meal:** Harness configured in `.claude/hooks.json`. One reference hook: `.claude/hooks/tdd_enforcer.py` (advisory PostToolUse, ~80 lines). Warns when `app/` or `agents/` edited without corresponding `tests/` edit. State journaled to `.claude/state/tdd_enforcer.jsonl`.
**When it runs:** Every time a Worker edits `app/routes/orders.py` without touching `tests/integration/test_orders.py`.
**Phase 2 seam:** `.claude/hooks.json` registry grows from 1 entry to N; harness and schema unchanged.

### P11 — Self-Validation Loop (3-step lite)
**Phase 1 application in ds-meal:** Doc convention only, no hook. 3-step lite: `State claim → Steel-man → Score (≥80%)`. In `docs/systems/self-validation-loop/SVL-ARCHITECTURE.md`. Required in every PRP and at the bottom of architecture docs making HIGH-magnitude claims.
**When it runs:** Reviewer opens a PRP and sees the SVL box at the bottom: claim stated, steel-manned, scored 85%. No runtime component.
**Phase 2 seam:** SVL doc adds Prove + Measure steps; existing 3-step usages stay valid (subset).

### P12 — Reviewable Artifact + Phase Gate
**Phase 1 application in ds-meal:** Doc convention. Artifact Rule: commit and push any file before quoting its path. Phase Gate: between phases run `Save → Merge → Push → Confirm`. Documented in `CLAUDE.md`.
**When it runs:** After Phase 2 completes, the 4-step gate fires: `git add docs/workflows/PROTOCOL-APPLICATION-MATRIX.md && git commit && git push && confirm`. No Phase 3 work begins until push succeeds.
**Phase 2 seam:** Port DuloCore's `.claude/hooks/artifact_enforcer.py` into ds-meal's `.claude/hooks.json`.

### P13 — Rollout Pattern (PoC first)
**Phase 1 application in ds-meal:** Convention. Menu Planner first tested end-to-end on **one** facility (Riverside) before running against all five. NL Ordering first validated on 5 seed phrases in `tests/fixtures/nl_seed_phrases.json`. PRP template has a "PoC Gate" field.
**When it runs:** During demo, presenter shows Menu Planner working on Riverside, then explicitly *does not* demo it against Harbor View — Phase 2 scale-out.
**Phase 2 seam:** `tests/fixtures/nl_seed_phrases.json` grows to ~50 phrases; Playwright E2E adds parameterized coverage across 5 facilities.

### P14 — Frontend/Backend Decoupling (HTML + JSON twins)
**Phase 1 application in ds-meal:** Every HTML route in `app/routes/*.py` has a 1:1 JSON twin under `/api/v1/`. Both handlers call the same service function in `app/services/*.py`. Example: `GET /orders/{id}` and `GET /api/v1/orders/{id}` both call `app/services/orders.py::get_order_with_timeline(id)`. OpenAPI auto-generated at `/docs`.
**When it runs:** Reviewer hits `/orders/123` and sees Jinja timeline; reviewer hits `/api/v1/orders/123` and sees the same data as JSON.
**Phase 2 seam:** Jinja templates deleted; React app pointed at `/api/v1/*`. Zero backend changes.

### P15 — Authentication Isolation (separate Clerk app)
**Phase 1 application in ds-meal:** Dedicated Clerk application `ds-meal-prototype` (free-tier Development), zero shared state with any other Clerk tenant. Keys in `.env.ds-meal` (gitignored). `app/auth/clerk_middleware.py` ~40 lines JWKS verification. `app/auth/dependencies.py::require_login` gates routes. Allowlist: single row `Facility.admin_email = admin@dulocore.com` on Riverside. Double-check at `/sign-in/callback`: JWT email vs `Facility.admin_email` → 403 on mismatch.
**When it runs:** User clicks "Sign in with Google" on `/sign-in`, Clerk hosts OAuth, callback hits `/sign-in/callback`, provisioning creates `User` row, redirect to `/facility/dashboard`.
**Phase 2 seam:** `app/auth/dependencies.py::require_login` extends to `require_role(role)` for multi-tenant RBAC.

### P16 — Design Principles (SOLID/DRY/KISS/YAGNI/12-factor)
**Phase 1 application in ds-meal:** Canonical principles doc adapted from DuloCore's `Docs/meta/SOFTWARE-ENGINEERING-PRINCIPLES.md` to `docs/systems/architecture-principles/ARCHITECTURE-PRINCIPLES.md`. Preserves the AHA nuance for DRY. 12-factor mapped concretely: config in `.env.ds-meal`, stateless app, SQLite file as bind-mounted volume, logs to stdout + `logs/`. Cited in PRPs and module docstrings.
**When it runs:** Reviewer opens the file during code review; cited inline (e.g., `agents/tools.py` vs `agents/tools_sdk.py` are two files because of SRP).
**Phase 2 seam:** `docs/systems/architecture-principles/CHECKLIST.md` — pre-PR checklist referencing each principle.

### P17 — Business Plan Architecture (memo-style WHY doc)
**Phase 1 application in ds-meal:** Memo saved to `docs/business/BUSINESS-PLAN-ARCHITECTURE.md`. Sections: target users, value prop, App Phase 1 scope vs Phase 2 roadmap, explicit non-goals, Staff-level engineering story.
**When it runs:** First thing an interviewer reads — linked from `README.md` and `docs/INDEX.md`.
**Phase 2 seam:** Add "Pricing & Unit Economics" section when payment integration lands.

### P18 — Deferred-with-Seam (explicit Phase 2 items)
**Phase 1 application in ds-meal:** Single "Phase 2 Roadmap" at `docs/PHASE-2-ROADMAP.md` naming every deferred item with its graduation seam: Inngest message bus; MiniLM embeddings; graph KB (Kuzu/Neo4j) for wiki gap detection; 7-point wiki lint; 24-hour compile cron; multi-tenant RBAC; payments/ERP/routing/mobile. Mirrors the Seam catalog in the Phase 1 plan.
**When it runs:** During demo Q&A when interviewer asks "what about X?" — presenter opens this doc and answers with the named seam.
**Phase 2 seam:** This doc IS the seam index. When an item graduates, its row moves from `PHASE-2-ROADMAP.md` to `CATALOG.md` as "Built."
